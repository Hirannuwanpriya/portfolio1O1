## Overview

The Lodge is a long-running restaurant and bar-club in Watermael-Boitsfort,
Brussels, operating across three revenue surfaces from one venue: a
Franco-Belgian brasserie restaurant, an upstairs bar-club, and a private
events business handling birthdays, corporate team building, and full
privatisations.

From an engineering perspective the brief was effectively three products on
one domain — a content-heavy restaurant site, a transactional reservation
flow, and a structured B2B enquiry funnel for events — all served bilingually
in French and Dutch. The challenge was to model these surfaces cleanly behind
one Laravel application, share brand and layout primitives without coupling
the channels, and keep the public front end fast enough to score in the
green on Core Web Vitals while serving rich venue photography.

## The Problem

- No existing data model — the menu (six sections), event packages,
  reservations, enquiries, and venue gallery all needed to be designed from
  scratch with relationships the client could maintain without engineering.
- No translatable content layer — the venue serves a bilingual Brussels
  audience but had no Dutch surface, limiting both discoverability and SEO
  reach for Flemish-speaking visitors.
- No transactional reservation surface — bookings were captured by phone
  with no online flow and no digital audit trail.
- No structured events enquiry pipeline — leads landed on a generic email
  inbox with no validation, no triage, and no required-field enforcement.
- No CMS — every menu and content change required developer involvement,
  which was unsustainable for seasonal menus.
- Performance and SEO budget — the site needed to rank in a competitive
  Brussels restaurant search market while serving high-resolution venue and
  food photography.

## My Role

Sole engineer across the full stack: schema design, Laravel application
architecture, Nova admin build, Vue and Alpine front-end implementation,
bilingual content layer, reservation and enquiry pipelines, AWS
infrastructure, Cloudflare CDN configuration, deployment pipeline, and
client handover including admin training documentation.

## Solution

### Architecture

A Laravel monolith serves the whole site as server-rendered HTML, with Vue.js
islands mounted only on the routes that need interactivity — the reservation
flow and the menu navigation/gallery. Lighter pages (events, contact, brand
story, opening hours) use Alpine.js for progressive enhancement, which keeps
the JavaScript payload off pages that don't need a SPA-style runtime. This
hybrid was a deliberate choice over a single Vue SPA: it preserves SSR for
SEO, simplifies cache invalidation across two locales, and avoids shipping
the reservation bundle to users reading the menu.

The data layer is a normalised MySQL schema with Eloquent relationships
covering menu sections → dishes, event packages → package items,
reservations, enquiries, gallery media, opening hours, and translatable
content per locale. A custom Laravel Nova admin exposes every entity to the
client with side-by-side FR / NL fields, inline media management, and soft
deletes.

The application runs on AWS EC2 behind Cloudflare, with a queue worker
handling outbound mail and image variant generation so the request-response
cycle for reservations and enquiries stays tight.

### Bilingual Content Layer

FR and NL are implemented as translatable Eloquent attributes on every
content model rather than parallel content trees. Each record carries
locale-aware fields (title, description, slug) on the same row, so
relationships, ordering, and IDs have a single source of truth. Locale-prefixed
routes (`/fr/...`, `/nl/...`) drive canonicals and hreflang emission per
indexable URL, and inside Nova the FR and NL fields are surfaced
side-by-side on the edit form so the client maintains parity without
navigating between sections.

### Restaurant Menu System

The full menu is managed through Nova across six sections: seasonal
suggestions, starters, salads, pasta, poultry, and meat. Sections are
reorderable and dishes are independently editable; the client updates
descriptions, allergen flags, and prices without engineering involvement.
Seasonal suggestions are surfaced as a priority block on the menu page so
the chef's market-led specials sit above the core carte. Dish records carry
locale-aware text fields so the FR menu (`/fr/notre-carte/`) and the NL
menu (`/nl/onze-kaart/`) stay in lockstep when the menu is updated.

### Reservation Flow

Reservations run through a dedicated `ReservationRequest` form request with
field-level validation — party size bounds, date in the future, opening-hours
check, required service slot — and Alpine mirrors the rules client-side for
instant feedback. A successful submission creates a `Reservation` record,
queues a confirmation email to the guest and a notification to the venue's
inbox, and surfaces the booking in Nova with status filters. Honeypot and
rate limiting on the route keep spam out without a captcha that would hurt
conversion.

### Events Enquiry Pipeline

Event packages (birthday, corporate, full privatisation) are modelled as
their own entity with package-level fields (per-person price, group size
range, included items) so each event format renders from data instead of
hardcoded copy. The enquiry form is the highest-value lead capture on the
site and runs through an `EventEnquiryRequest` with required event type,
date, party size, and contact fields. Successful submissions create an
`Enquiry` record, dispatch a queued notification to the events inbox, and
send the enquirer a templated confirmation in the locale they were browsing.

### Custom CMS via Laravel Nova

Rather than bolt on a generic CMS, I built the admin as Nova resources
mapped directly to the domain model. Menu sections are reorderable via
drag-and-drop, dishes carry structured allergen data, event packages have
their own resource with package-item nesting, and reservations and
enquiries surface in dashboards with filters by date, status, and event
type. The client handles every content change without engineering
involvement, which was the explicit success criterion at handover.

### AWS S3 + Cloudflare Media Pipeline

All gallery, menu, and venue imagery is uploaded through Nova, processed
into responsive variants by an `intervention/image` queued job, and stored
on S3. Cloudflare sits in front as the CDN with long-cache headers and
Polish enabled for additional optimisation. The front end requests images
through `srcset` with explicit `width`/`height` to eliminate CLS, and uses
native `loading="lazy"` below the fold — important on the gallery-heavy
homepage where venue photography is the primary conversion asset.

### Performance, SEO, and Accessibility

- Server-rendered HTML for every public route keeps TTFB low and gives
  crawlers fully populated pages, including translated content.
- Vue is mounted as islands only where interactivity is needed; the
  reservation bundle never loads on the menu or events pages.
- Tailwind is purged to the actually-used class set, dropping production
  CSS to a small fraction of the dev build.
- Structured data (Restaurant, Menu, LocalBusiness) is emitted from Blade
  partials so each entity in Nova drives its own schema.org payload, with
  per-locale `inLanguage` annotations.
- hreflang and canonical URLs are emitted per indexable URL.
- Forms, navigation, and the reservation flow are keyboard-navigable with
  visible focus rings and ARIA labelling on icon-only controls.

### Key Features

- Server-rendered Laravel front end with Vue.js islands for the reservation
  flow and menu interactions, Alpine.js for lighter pages
- Normalised MySQL schema with translatable Eloquent attributes across every
  content model
- Six-section menu CMS with reorderable sections, allergen flags, and
  side-by-side FR / NL editing in Nova
- Validated reservation pipeline with server + client validation, queued
  confirmations, honeypot, and rate limiting
- Validated events enquiry pipeline with structured event packages and
  locale-aware confirmation mail
- Queued image processing pipeline producing responsive variants on S3,
  served through Cloudflare with optimisation enabled
- schema.org structured data emitted per entity with per-locale `inLanguage`
  annotations
- Google Maps and Facebook integrations lazy-loaded so they never block
  initial render

## Outcomes

- One Laravel codebase cleanly serves three revenue surfaces — restaurant,
  bar-club, and events — with no shared state leaking between them, and the
  client maintains all content through Nova without engineering involvement.
- The bilingual content layer extends discoverability to Flemish-speaking
  Brussels diners with full hreflang and canonical coverage, served from a
  single content model rather than parallel trees.
- The reservation pipeline captures bookings end-to-end with validation,
  audit trail, and queued confirmations — replacing a phone-only intake.
- The events enquiry pipeline lands structured leads with required fields
  and locale-aware confirmation, removing the back-and-forth that previously
  preceded every event quote.
- Public pages score in the green on Core Web Vitals; the JS payload on
  content pages stays under ~80KB by keeping Vue out of routes that don't
  need it.
- The admin model is documented and handed over so the venue, kitchen, and
  events teams each see only their own surface.

## Tech Stack Detail

**Laravel** is the backbone — routing, controllers, Eloquent models, form
requests, jobs, notifications, and the queue. Domain logic (reservation
validation, enquiry routing, locale resolution) lives in service classes
invoked from controllers, and the queued mail and image jobs are
independently retryable.

**Vue.js** powers the interactive islands: the reservation flow (with
reactive availability + validation feedback) and the menu navigation /
gallery carousel. Each island is mounted to a single DOM node with its own
bundle, so routes that don't need them never download them.

**Alpine.js** handles progressive enhancement on lighter pages — the mobile
navigation drawer, the events enquiry form's client-side feedback, and the
opening-hours expandable block — keeping these routes essentially
zero-runtime.

**Laravel Nova** is the admin layer. Every domain entity has a Nova resource
with side-by-side FR / NL fields, custom actions (drag-and-drop section
reordering, status transitions on reservations), and lenses for the events
team's enquiry triage view.

**Livewire 3** is used for the admin-side reservation and enquiry
dashboards where reactive filtering and status updates benefit from
server-driven state without a client SPA.

**MySQL** holds the normalised schema. Indexes are placed on the foreign
keys that drive the public menu and gallery queries, and reservation
lookups are indexed by date for fast availability checks.

**AWS S3 + Cloudflare CDN** form the media pipeline — uploads land on S3,
a queued image job produces responsive variants, and Cloudflare serves
them with long cache TTLs and Polish optimisation.

**Google Maps API** is embedded for the location surface, lazy-loaded so
it never blocks the initial render.

**Facebook Integration** surfaces the venue's social feed without
loading their SDK above the fold.

## Challenges and Learnings

The most interesting engineering decision was the channel boundary. The
business wanted three products on one domain in two locales; the easy path
was a single Vue SPA hitting a Laravel API, but that would have meant
shipping every channel's JavaScript to every visitor, fighting SSR for
bilingual SEO, and inventing a cache invalidation story across two
languages. Choosing a server-rendered monolith with Vue islands kept each
channel cheap to load, kept SEO simple per locale, and kept the admin model
tightly coupled to the routes that consume it.

The translation layer taught me to prefer translatable attributes over
parallel content trees on this kind of multi-locale catalogue. A single
source of truth for relationships, ordering, and slugs kept the admin
ergonomics clean — the client maintains one menu record with FR and NL
fields, not two records that have to be kept in sync by hand. Surfacing
both locales side-by-side in the Nova edit form was the small UX detail
that made parity actually achievable in practice.

The reservation and enquiry pipelines reinforced two patterns I now reach
for by default on any form-driven feature: always validate twice (server
form requests are the authority, Alpine mirrors the rules for instant
feedback), and always queue side-effects behind a retryable job. The first
prevents tampering and zero-state bugs; the second makes mail delivery
hiccups invisible to the user instead of fatal to the request.

The decision to keep the Facebook and Maps integrations lazy-loaded looks
small but mattered for Core Web Vitals — the homepage is gallery-heavy by
design, and pulling third-party SDKs above the fold would have eaten the
performance budget that the venue photography needed.
