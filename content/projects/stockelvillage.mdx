## Overview

Stockel Village is the digital platform for a major commercial district in
Brussels, covering 250+ retail shops, restaurants, personal services, a cinema
(Le Stockel), and a year-round programme of community events. Engineering-wise
the brief was three problems on one codebase: a legacy data extraction with no
database access, a relational rebuild of the public directory and CMS, and a
scheduled membership billing engine that had to run unattended every month
across multiple locales.

I was the sole engineer across the full stack — migration tooling, schema
design, Livewire components, Nova admin, queue pipeline, Horizon configuration,
deployment via Forge + Envoyer, and Cloudflare cache rules.

## The Problem

- Legacy WordPress install with no shell, no database credentials, and no
  developer handover — the only access surface was an admin username and
  password, so the schema had to be reverse-engineered from the inside.
- No automated billing — 250+ memberships were tracked by hand, with
  reminders, overdue follow-ups, and suspensions all happening over email.
- No translatable content layer — the FR / NL / EN audience was being served
  one language despite living in a tri-lingual city.
- Read-heavy directory routes with category filters, full-text search, and
  merchant profile pages had to stay fast under sustained traffic and grow to
  4× the current listing count without re-architecture.
- Real-time publishing — the association wanted approved events on the
  homepage feed instantly, not on the next deploy.

## My Role

Sole engineer across the full stack: legacy data extraction tooling, Laravel
schema design, Livewire 3 component build, Laravel Nova 5 admin, the
membership state machine, Horizon queue tuning, FR / NL / EN translation
layer, performance work, and deployment pipeline (Forge + Envoyer behind
Cloudflare). Reverse-engineering the WordPress data model came first — no new
code was written until every legacy entity was mapped to a target Eloquent
model.

## Solution

### Architecture

A Laravel 11 monolith with Livewire 3 driving the interactive surfaces
(directory filters, search, admin dashboards) and Alpine.js handling lighter
interactions on content pages. The data layer is a normalised MySQL schema
covering merchants, categories, opening hours, events, cinema screenings,
memberships, invoices, suspensions, locales, and media — all wired through
Eloquent relationships and exposed in Laravel Nova 5 as the editorial admin.

Asynchronous work runs through Laravel Horizon: image processing, outbound
mail, the scheduled monthly billing job, and webhook-style event broadcasts
via Laravel Echo. The app is deployed via Forge + Envoyer for zero-downtime
releases, fronted by Cloudflare for static asset caching and S3 for media.

### Phase 1 — WordPress Reverse-Engineering and Migration

With no direct database access I treated the legacy site as an opaque API.
Every WP admin section was walked methodically — custom post types, ACF
field groups, taxonomy hierarchies, user roles, media library, and plugin
tables — and each entity was documented and mapped to a target Eloquent
model before extraction began.

A custom Artisan command pulled records via the WordPress REST API and
admin-accessible XML exports, normalised inconsistent category assignments
and duplicate slugs, rebuilt the foreign-key graph (merchant → category,
merchant → opening hours, merchant → media, merchant → locale variants),
and seeded the new schema. Data that was not reachable through any API
surface was flagged for manual entry early so there were no go-live gaps.

### Phase 2 — Directory, Events and Cinema

The merchant directory is a Livewire 3 component with debounced full-text
search, multi-select category filters, and merchant profile pages carrying
opening hours, contact, gallery, and event associations. Eager loading and
composite indexes on the foreign keys that drive filter queries keep the
listing route fast under sustained traffic.

Le Stockel cinema lives as a sibling content type with its own Eloquent
model and Nova resource — separate from the merchant listing — so programme
schedules and screening data are managed independently and never collide
with merchant-level fields.

### Phase 3 — Membership State Machine

Membership is modelled as an explicit four-state machine — active,
reminded, overdue, suspended — with every transition appended to a
`membership_events` table for audit. The flow:

1. A scheduled Laravel job runs on the first of every month and is
   dispatched onto Horizon.
2. For each eligible merchant the job resolves current payment status and
   transitions the state machine.
3. Localised reminder mail (FR / NL / EN) is queued through Mailgun in the
   merchant's preferred locale.
4. Idempotency is keyed on `(merchant_id, billing_cycle)` so retries and
   re-runs never double-send.
5. After three consecutive overdue cycles the merchant flips to
   `suspended` and a global Eloquent scope drops them from public queries.
6. The suspension email carries a signed reactivation URL that restores the
   listing without admin involvement.

### Phase 4 — Multi-Language Content Layer

FR / NL / EN are implemented as translatable Eloquent attributes on every
content model rather than parallel content trees, keeping a single source of
truth for relationships, ordering, and IDs. Locale-prefixed routes
(`/fr/...`, `/nl/...`, `/en/...`) drive canonicals and hreflang emission per
merchant URL. Inside Nova the locale fields are surfaced side-by-side on the
edit form so editors can keep parity between languages without leaving the
record.

### Performance and Caching

- Directory and merchant routes are cached at the Cloudflare edge with
  short TTLs and tag-based purges fired from Eloquent observers on save.
- Hot queries are eager-loaded; composite indexes back every filter
  combination on the directory listing.
- Image uploads are queued through Horizon for variant generation and stored
  on S3 with long-cache headers and explicit `width`/`height` to eliminate
  CLS on listing pages.
- Tailwind is purged to the actually-used class set, dropping production
  CSS to a small fraction of the dev build.

### Real-Time Event Publishing

Approved events broadcast via Laravel Echo so the homepage feed updates
without a deploy or a page reload — an Eloquent observer fires a broadcast
event on `Event::approved`, and a small Alpine listener prepends the new
card to the feed.

### Key Features

- Livewire 3 directory with full-text search, category filters, and
  merchant profile pages backed by a normalised MySQL schema
- Laravel Nova 5 admin covering 14 entities with role-scoped access and
  inline media management
- Scheduled monthly membership job on Horizon driving a four-state machine
  with idempotent transitions and audit trail
- FR / NL / EN translatable content across every model, with
  locale-prefixed routes and hreflang emission
- Cloudflare-edge caching on directory routes with tag-based purges from
  Eloquent observers
- Real-time event broadcasting via Laravel Echo to the homepage feed
- Forge + Envoyer zero-downtime deployment pipeline
- Custom Artisan migration command that ported the legacy WordPress data
  set without database access

## Outcomes

- All 250+ merchant records migrated from WordPress with relational
  integrity intact and zero go-live data loss.
- Directory routes sustain 10,000+ monthly visitors with consistent
  uptime since launch, served behind Cloudflare with zero-downtime
  releases via Envoyer.
- Membership engine runs fully unattended every month — no admin
  intervention required unless a merchant initiates contact.
- Schema, indexes, and query paths sized for 1,000 merchant listings (4×
  current volume) without infrastructure changes.
- Full FR / NL / EN translation live across the public site and the Nova
  admin, with proper canonicals and hreflang per locale.
- Approved events publish to the homepage feed in real time via Laravel
  Echo, removing the wait between admin approval and visibility.

## Tech Stack Detail

**Laravel 11** is the backbone — routing, controllers, Eloquent models,
form requests, scheduled jobs, notifications, and console commands. The
membership state machine lives in a dedicated service class and is invoked
both by the scheduled job and by Nova actions for manual overrides.

**Livewire 3** powers the interactive directory components — filters,
search, and the admin dashboards — keeping the public surface server-driven
while still feeling reactive.

**Laravel Nova 5** is the admin layer. Every domain entity has a Nova
resource with custom fields and actions, and the membership engine surfaces
its audit trail through a dedicated lens.

**Horizon** supervises queues for the monthly billing job, Mailgun
deliveries, image processing, and Echo broadcasts — with retry and
back-off policies tuned per queue.

**Laravel Echo** broadcasts approved events to a homepage channel so the
feed updates without a deploy or a manual refresh.

**Alpine.js** handles lightweight client interactions — the mobile nav,
expandable opening-hours blocks, and the inline cookie banner.

**Tailwind** is the utility layer; purged to the used class set in
production.

**AWS EC2 + S3 + Cloudflare** form the hosting and media stack. **Forge**
provisions and configures the server; **Envoyer** runs the zero-downtime
release pipeline.

## Challenges and Learnings

The most interesting engineering decision was treating the legacy
WordPress install as an opaque API rather than fighting for database
access. Walking the admin systematically and modelling every entity before
writing any new code took the first two weeks of the project, but it
turned the migration from a guessing game into a deterministic port — by
the time the Artisan command ran end-to-end, every legacy field had a
target Eloquent attribute and every relationship had a foreign key.

The membership engine reinforced the value of explicit state machines for
scheduled work. Modelling membership as four named states with a typed
event log made the monthly job trivially auditable: any merchant's status
on any given month is reconstructable from the `membership_events` table,
and idempotency keyed on `(merchant_id, billing_cycle)` made Horizon
retries a feature rather than a hazard.

The translation layer taught me to prefer translatable attributes over
parallel content trees on this kind of directory. A single source of
truth for relationships, ordering, and slugs kept the admin ergonomics
clean — editors maintain one merchant record with three locale fields,
not three records that have to be kept in sync by hand.

The Cloudflare cache tagging pattern — Eloquent observers firing
tag-based purges on save — was the single biggest win for read
performance. The directory route serves the cached HTML for the vast
majority of requests, and editors see their changes propagate within
seconds without us having to invalidate the whole zone.
