## Overview

A trustee real estate platform managing property portfolios on behalf of
owners, with a non-negotiable product requirement: every view that surfaces
money must keep owner funds and manager funds strictly separated and
independently auditable. I joined as the dedicated front-end engineer,
working from Figma designs against a MongoDB-backed REST API built in
parallel by a France-based back-end developer. Over eight months I built
the Vue.js component architecture, integrated the full REST surface, wired
the S3 document pipeline, and shipped the application end-to-end against
the design and the live contract.

## The Problem

- The product surface spans property records, tenancy state, financial
  ledgers, and trustee documents — every view needed a consistent component
  vocabulary, not a per-screen one-off.
- Fund separation between owner and manager accounts had to be enforced
  architecturally, not just visually — a single component quietly mixing
  the two would break the product's core promise.
- The REST API was being built in parallel by a remote back-end developer,
  so front-end integration could not block on endpoints existing yet.
- Document handling (leases, statements, trustee reports) needed direct-to-S3
  uploads with progress reporting, content-type validation, and inline
  preview — not a naive file input posted through the application.
- The audience is non-technical trustees and property owners, so navigation,
  table density, and financial layouts had to stay clear without compromising
  on the data they came to the screen to see.

## My Role

Sole front-end engineer for the duration of the project. I owned the Vue.js
project setup, the component architecture, the design system mapped to
Figma tokens, the REST integration layer (Axios client, store modules,
contract verification through Postman), the S3-backed document pipeline,
the responsive layout, accessibility, QA, and delivery. The back-end was
owned by a France-based developer; the contract handshake between us was
itself a deliverable.

## Solution

### Architecture

A Vue.js single-page application built around a Vuex-style store with one
module per domain — properties, tenancies, owner ledger, manager ledger,
documents, users — each consuming the REST API through a shared, typed
Axios client. Tailwind CSS drives the styling system, with a token set
mapped directly from the Figma file so spacing, colour, and typography
stay synchronised between design and implementation. The design system
ships as a library of ~25 primitives that compose into every property and
finance view.

The application is hosted on AWS EC2 provisioned through Forge, fronted
by Cloudflare for caching and TLS. Static assets and trustee documents
live on S3, with signed URLs generated by the back-end for both upload
and read access.

### Component Architecture

The component library is layered:

1. Primitives — buttons, inputs, badges, modals, table cells.
2. Compositions — data tables, ledger rows, fund summary cards, document
   uploaders, navigation shells.
3. View components — property detail, tenancy panel, owner ledger,
   manager ledger, document list.

Every view is built from compositions; compositions are built from
primitives. A new screen adds a view, not a new primitive — which keeps
the visual language tight and gives QA a small set of components to
verify exhaustively rather than a sprawling per-view surface.

### REST Integration and Contract Verification

The Axios client is the single audited surface for every API call. It
handles auth tokens, response unwrapping, error normalisation (a single
typed error shape regardless of which endpoint failed), and retry on
idempotent reads. Each store module consumes it through a typed response
interface that mirrors the Postman-verified shape exactly.

The Postman collection is the contract handshake with the back-end. Every
endpoint has its request and response shape captured there; schema
changes are proposed against the collection and reviewed in writing
before either side merges. When an endpoint isn't live yet, components
are built against mocks generated from the Postman response, so the
real integration is a one-line swap on the client base URL.

### Fund Separation as an Architectural Invariant

Owner and manager funds are not just colour-coded — they are isolated at
every layer. Each ledger has its own Vuex module, hits its own API
namespace, and renders through a single shared `FundBadge` primitive that
takes the fund type as a prop. The `FundBadge` is the only component in
the system allowed to render a fund-attributed amount; a ledger row
without one fails design review. A store module reaching across into the
other ledger's namespace is blocked at code review. The architecture
makes leakage impossible, and the visual language — colour, position,
label — sits on top of that guarantee rather than substituting for it.

### Document Pipeline

Trustee documents go through a direct-to-S3 upload using pre-signed URLs
issued by the back-end. The front-end handles content-type validation
client-side (PDF, image formats, common office types), progress reporting,
graceful error handling on network failure, and inline preview for the
document types trustees actually deal with. The application server never
proxies file bytes — uploads go straight from the browser to S3, and
reads come through signed URLs with short TTLs.

### Responsive and Accessible UI

- Tables degrade to stacked card layouts at tablet widths without losing
  column alignment, so trustees reviewing portfolios on the move never
  see a broken row.
- Every interactive element is keyboard-navigable with visible focus
  rings; modals trap focus and restore it on close.
- Form fields surface validation inline with `aria-describedby` wiring,
  not just colour.
- Tailwind is purged to the production class set, keeping the CSS
  payload small.

### Key Features

- Vue.js SPA built around domain-scoped store modules, each consuming a
  typed Axios client wrapping auth, error normalisation, and retry
- Component library of ~25 primitives mapped to Figma tokens, composed
  consistently across every property and finance view
- Architecturally enforced fund separation — owner and manager ledgers
  live in separate stores, fetch from separate API namespaces, and render
  through a single shared `FundBadge` primitive
- Postman-verified contract handshake with the back-end — schema changes
  proposed against the shared collection before either side merges
- Direct-to-S3 document uploads via pre-signed URLs with content-type
  validation, progress reporting, and inline preview
- Responsive layout down to tablet, with tables that degrade to stacked
  cards without losing data alignment
- Forge-deployed on AWS EC2 behind Cloudflare for caching and TLS

## Outcomes

- The full product surface shipped to Figma against 40+ live REST
  endpoints, with no silent contract mismatches between front and back
  ends because every shape was verified in Postman first.
- Fund separation became a structural property of the application, not a
  visual convention — which means future feature work cannot accidentally
  violate the product's core promise.
- A single typed Axios client centralised auth, retries, and error
  shapes, so every store module consumes the API through one audited
  surface rather than scattered fetch calls.
- The component library compresses 25+ primitives into the entire
  product surface, keeping the visual language tight and giving QA a
  finite set of components to test exhaustively.
- Eight months of cross-timezone collaboration with the back-end
  developer ran without stalls, because the Postman collection was the
  source of truth and mock-first integration unblocked the front-end
  whenever an endpoint was still in flight.

## Tech Stack Detail

**Vue.js** is the application framework. Single-page architecture with
domain-scoped store modules, view components composed from a shared
primitive library, and a typed Axios client as the single integration
surface.

**Tailwind CSS** drives the styling system, configured against a token
set extracted from the Figma file so spacing, colour, and typography
stay synchronised between design and implementation.

**MongoDB** is the back-end's document store. The front-end consumes it
through a REST API; the document model shapes the response payloads but
the front-end stays agnostic of Mongo specifics.

**REST API** integration goes through a single Axios client handling
auth, response unwrapping, error normalisation, and retry. Every call is
typed against a Postman-verified shape.

**AWS EC2 + AWS S3 + Cloudflare** form the hosting and delivery layer —
EC2 (provisioned via Forge) for the application, S3 for trustee
documents and static assets, Cloudflare in front for caching and TLS.

**Forge** provisions and deploys the host.

**Figma** is the design source of truth. Tokens extracted from the file
drive the Tailwind config so design and implementation never diverge.

**Postman** is the API contract. Every endpoint has its request and
response shape captured in the shared collection, and schema changes
are proposed against it before either side merges.

## Challenges and Learnings

The most useful engineering pattern from this project was treating the
Postman collection as the contract handshake. Cross-timezone, two-person
teams break down when one side ships against a stale assumption about
what the other has implemented. Making the contract a written artefact
that both sides edit together — and treating mocks generated from that
contract as a first-class development surface — meant the front-end was
never blocked on endpoints existing, and integration day was a no-op
swap rather than a discovery exercise.

Enforcing fund separation architecturally rather than visually was the
other lesson worth keeping. Visual conventions get violated; structure
does not. Once the owner and manager ledgers lived in separate store
modules, separate API namespaces, and rendered only through a shared
`FundBadge`, the only way to leak data across them was to deliberately
break the architecture — which a code review catches in a way that a
missing colour swatch does not.

The typed Axios client paid back compounding interest over eight months.
A single audited surface for auth, error shape, and retry meant that
every new feature inherited the same guarantees, and that bugs in API
handling were fixed in one place rather than scattered through fetch
calls. On any future SPA project this is the first thing I'd build.

The Figma-to-Tailwind token pipeline kept design and implementation
honest. Pulling spacing, colour, and typography from one source meant
the design system stayed tight even as the surface grew, and the
component library could be extended without re-deriving values from
screenshots.
