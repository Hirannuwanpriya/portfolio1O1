## Overview

A laundry business in France was running its entire point-of-sale and order
management on a single MS Access installation tied to one physical workstation.
The application was functional but had no relational integrity, no role
separation, no automated customer notifications, no remote access, and no
recovery path if the machine failed. The brief was to rebuild it as a modern
web platform on Laravel and React/Inertia — preserving every historical record
and every business rule the operator already relied on.

## The Problem

- The data layer was an MS Access `.mdb` file with implicit relationships,
  loose typing, and years of accumulated inconsistencies that a real RDBMS
  would reject — duplicate customer entries, nullable foreign keys, and
  free-text fields used as enums.
- All business logic lived in Access forms, queries, and VBA macros, with no
  schema documentation and no separation between data, behaviour, and UI.
- Every staff member had unrestricted access to the entire database — pricing
  configuration, historical revenue, customer records — with no role boundary.
- Customers had to be phoned manually when an order was ready; there was no
  notification surface and no order status machine to drive one.
- The application could only be used on one workstation. There was no path
  to a second terminal, let alone a second site or remote access.

## My Role

Sole engineer across the full stack: schema reverse-engineering, Laravel
domain design, Inertia/React frontend, RBAC layer, notification pipeline,
artisan-based data migration, and AWS deployment via Forge. Discovery and
contract verification were a formal deliverable in their own right — every
business rule was confirmed in writing with the client before code shipped.

## Solution

### Architecture

A Laravel monolith serves the application via Inertia, with React + TypeScript
on the client side. Pages are server-rendered shells hydrated by Inertia,
which means the routing, auth, and permission model live entirely in Laravel
while interactive surfaces (order intake, the customer panel, pricing rules)
ship as React components. Alpine.js handles light progressive-enhancement
behaviours on report and settings pages where a full React island would be
overkill. Tailwind drives the styling system end-to-end.

The data layer is a normalised MySQL schema with proper foreign keys, enum
columns for order status and role, and indexes on the lookups that drive the
hot screens — order list by status, customer by phone, transactions by date
range. Redis backs queues, cache, and session storage. Files (receipts,
photos of damaged garments, exported reports) live on S3, fronted by
Cloudflare for delivery.

The application runs on AWS EC2 provisioned through Laravel Forge, with a
queue worker dedicated to notification jobs and a separate worker for
report generation so a slow PDF export never blocks an SMS dispatch.

### Domain Modelling and RBAC

Six domain modules — orders, customers, pricing rules, notifications,
reporting, RBAC — each own their controllers, form requests, policies,
service classes, and Inertia pages. Authorization is enforced two ways: a
Laravel policy on every controller method (so a Staff user calling a Manager
endpoint gets a 403 even if they craft the request manually) and an Inertia
shared-props payload that exposes the current role to React, so the UI hides
controls the user cannot use. Pricing rules and end-of-day revenue are
gated to Owner; intake and status transitions are open to Staff; customer
profile editing is Manager-and-up.

### Order State Machine and Notification Pipeline

The order lifecycle has four explicit states — `received`, `in_progress`,
`ready_for_collection`, `collected` — modelled as an enum on the orders
table with transition rules enforced in a service class. Each transition
fires an Eloquent model event that dispatches a queued job per side effect:
a Twilio SMS job, a Mailgun email job, and a print job for the in-shop
receipt. Jobs are idempotent — keyed by `(order_id, target_status,
channel)` with a Redis lock — so retries after a worker crash never
double-notify a customer. Every dispatched notification is persisted, so
the operator can audit exactly what fired for any order from the customer
profile view.

### Dynamic Pricing Rules

Pricing is a first-class entity, not a hard-coded matrix. The owner
configures garment categories, modifiers (express, fragile, allergen-safe
detergent), and tiered customer discounts through the admin UI. The order
intake screen calls a pricing service with the selected garments and
modifiers and gets back a deterministic line-item breakdown that is
re-computed server-side at order save — the React client never gets to
write its own totals. Form requests validate every input against the
configured pricing rules so a tampered request cannot manufacture a price.

### Audited Migration via Artisan

Historical data lived in the `.mdb` file with the inconsistencies that
years of Access tolerance produced. The migration is a custom artisan
command that reads the file via ODBC, projects each row through the new
schema's validation rules, and routes failures to a structured review log
rather than skipping them silently. The client received the log with each
flagged record explained in plain terms, agreed corrections row by row,
and only then did the final import run. The migration is fully replayable
and the audit trail is preserved alongside the production database.

### Reporting

End-of-day reports were the operator's daily anchor in the legacy system,
so the new reports had to match the legacy totals and groupings exactly.
Each report is a Laravel class with a deterministic query and a snapshot
PDF rendered off a queued job, stored on S3 and surfaced in the React UI
with a signed download URL. Reports can be regenerated for any historical
date so the operator can reconcile against the legacy system during the
parallel-run period.

### Performance, Validation, and Accessibility

- Every write path goes through a Laravel form request — there is no
  controller method that trusts a raw payload.
- React forms surface server-side validation errors via Inertia's error
  bag without page reloads.
- Hot screens (order list, intake) ship as code-split React bundles so the
  initial paint stays fast on the in-shop terminal.
- Keyboard-first interaction patterns on intake and the customer panel —
  staff drive the system from the keyboard during peak hours.
- Tailwind is purged to the actually-used class set; the production bundle
  ships only what the UI uses.

### Key Features

- Six-module Laravel + Inertia/React app — orders, customers, pricing,
  notifications, reporting, and RBAC — each a discrete domain with its own
  controllers, policies, and React pages
- Three-tier role-based access control enforced in Laravel policies and
  mirrored to the React UI through Inertia shared props
- Order state machine with four lifecycle states firing idempotent queued
  Twilio SMS and Mailgun email jobs off Eloquent model events
- Dynamic pricing rules engine with server-authoritative line-item
  computation and form-request validation on every input
- Audited artisan migration from the legacy `.mdb` file with row-level
  validation, a client-reviewed reconciliation log, and zero data loss
- Report layer matching legacy end-of-day formats, rendered to PDF off a
  queued job and stored on S3 with signed URLs
- AWS EC2 + Forge deployment with Redis-backed queues, S3 file storage,
  and Cloudflare in front

## Outcomes

- A single-workstation Access install was replaced by a cloud-deployed,
  multi-user Laravel + React platform with no single point of failure.
- 100% of historical orders and customers were migrated with a documented
  audit trail; reconciliation against the legacy system during parallel
  run produced zero discrepancies.
- Manual collection calls were replaced by an event-driven Twilio + Mailgun
  pipeline; idempotent jobs survived the retries that any real notification
  pipeline has to tolerate.
- Pricing, reporting, and historical revenue became Owner-scoped through
  policy classes, so daily operation by Staff no longer exposed the
  business's commercials.
- End-of-day reports matched the legacy totals and groupings exactly,
  giving the operator continuity from day one.

## Tech Stack Detail

**Laravel** is the backbone — routing, controllers, Eloquent models, form
requests, policies, queued jobs, and the notification system. Each domain
module owns its slice of the application without leaking into the others.

**React.js + Inertia.js** drive the client. Inertia keeps routing and
authorization in Laravel while React renders the interactive screens —
order intake, customer panel, pricing rules — as code-split bundles.

**Alpine.js** handles light progressive enhancement on settings and report
pages where a full React island isn't justified.

**Tailwind CSS** drives the styling system, purged to the actually-used
class set in production.

**MySQL** holds the normalised schema. Foreign keys, enum columns, and
targeted indexes on the lookups that drive the hot screens.

**AWS EC2 + AWS S3 + Cloudflare** form the hosting and delivery layer —
EC2 provisioned through Forge for the app, S3 for receipts, photos, and
exported reports, Cloudflare in front for caching and TLS.

**Forge** provisions the server, deploys on push, and manages the queue
worker and scheduler.

**Redis** backs queues, cache, and idempotency keys for the notification
pipeline.

**Twilio** handles SMS notifications via a queued job per transition,
keyed for idempotency.

**Mailgun** handles transactional email — order receipts, status
updates, and end-of-day report deliveries.

## Challenges and Learnings

The most interesting engineering challenge was treating the legacy `.mdb`
as the source of truth without inheriting its problems. A relational
schema reverse-engineered from a non-relational database is only as
trustworthy as the validation layer that gates the import — and that
validation layer has to fail loudly, not silently. Routing every
inconsistent row to a structured review log instead of skipping it
turned the migration from a black box into a deliverable the client
could audit and sign off.

Designing the order state machine taught me to plan for retries from
day one. The instinct on a small project is to call Twilio inline from
the controller; the right pattern is one queued job per side effect,
keyed for idempotency. Once the lock and the keying scheme are in
place, retries become a feature — the worker can crash, the network
can blip, and the customer still gets exactly one SMS.

Building under a strict role boundary surfaced a subtle but important
pattern: authorization belongs in Laravel policies, but the UI has to
know about it too. Exposing the role through Inertia shared props let
the React layer hide controls the user can't use without ever
duplicating the policy logic — the policy stays the source of truth,
and the UI just listens.

The async, written-only client communication discipline ended up being
as valuable as any framework choice. Translating each module spec into
an annotated email forced clarity that real-time conversation often
glosses over, and meant that when development began every business
rule was already verified.
