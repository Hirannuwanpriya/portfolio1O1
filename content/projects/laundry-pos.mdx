## Overview

A laundry business in France had been running its entire point-of-sale and order
management operation on a Microsoft Access database installed on a single
workstation. The system was functional but fragile — one hardware failure away
from a complete operational halt — and had no way to notify customers
automatically, no role separation for staff, and no path to a second location.

I was engaged to rebuild the system as a modern web platform. The catch: the
Access application was entirely in French, the client spoke no English, and
there was no documentation, no handover package, and no colleague to ask.

## The Problem

- The entire business ran on a single MS Access installation on one machine.
  Any hardware failure would bring operations to a complete stop.
- Staff had to phone every customer manually when an order was ready for
  collection — a significant time drain with no fallback if calls were missed.
- Every staff member had unrestricted access to all records, including pricing
  configuration and full historical data.
- The system could not support a second location or any form of remote access.
- The Access database, application layer, and all business logic were written
  entirely in French, with no English-speaking contact available at the client.

## My Role

Sole engineer responsible for requirements discovery, system architecture,
frontend and backend development, data migration, and deployment. This project
required as much structured communication discipline as technical execution —
the discovery and verification phase was a formal project deliverable in its
own right, not a preliminary step.

## Solution

### Phase 1 — Legacy System Study and Translation

Before writing a line of new code, I worked through the entire Access file
systematically. Every form, query, report, and table was translated from French
into a structured English specification. Business logic embedded in Access macros
and VBA was documented and mapped to equivalent behaviour in the new system.

Each module spec was sent to the client as a detailed email with annotated
screenshots of the original Access screens alongside my interpretation of the
logic. The client replied in French confirming or correcting my understanding.
I translated each reply, updated the spec, and moved to the next module. No
development began until every module was client-verified.

### Phase 2 — Architecture and Development

Laravel service layer behind a Vue.js + TypeScript frontend. MySQL database
with a schema designed to mirror the verified Access data model while resolving
its structural weaknesses — normalising denormalised tables and introducing
proper relational constraints.

### Key Features

- Order intake with garment category selection, dynamic pricing rules, and
  expected ready date calculation
- Automated SMS and email notifications triggered on each order status change:
  received, in-progress, ready for collection, collected
- Customer profiles with full order history, notes, and visit frequency tracking
- Role-based access control with three levels — Owner, Manager, and Staff —
  each with scoped permissions across order management, pricing, and reporting
- End-of-day summary reports matching the format the owner was familiar with
  from the legacy Access system
- Cloud-hosted and accessible from any device on the shop network or remotely

### Phase 3 — Data Migration

All historical order and customer records were migrated from the Access `.mdb`
file via a custom Laravel artisan command using ODBC. A validation layer checked
every row against the new schema before writing, logging any record with
inconsistencies for client review rather than importing silently. Corrections
were agreed with the client before the final import ran, resulting in a clean,
fully auditable migration baseline.

## Outcomes

- Eliminated the single-point-of-failure dependency on one physical workstation.
- Reduced manual customer collection calls to near zero through automated SMS
  and email notifications.
- Introduced role-level visibility for the owner across staff activity and daily
  revenue without manual reconciliation.
- Migrated all historical order and customer data from Access with zero data loss.
- Delivered a system the client could use immediately, with reports and workflows
  matching the patterns they already knew.
